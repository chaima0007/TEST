## 3. Stratégie de revendications (claim hardening)

### Principe des 3 niveaux de protection

**Niveau 1 — Revendications indépendantes larges**
- Une revendication méthode (processus algorithmique)
- Une revendication système (architecture logicielle/matérielle)
- Une revendication produit programme d'ordinateur (support lisible)

**Niveau 2 — Revendications dépendantes spécifiques**
- Décliner chaque revendication indépendante en 5-8 variantes
- Couvrir : paramètres spécifiques, modes d'application, intégrations tiers

**Niveau 3 — Continuation après publication**
- Déposer 2-3 continuations dans les 18 mois suivant la publication
- Adapter les revendications à l'évolution du marché observée

### Exemples de contournements prévisibles et contre-mesures

| Contournement prévisible | Contre-mesure dans les revendications |
|--------------------------|---------------------------------------|
| Concurrent utilise autres poids | Revendiquer la méthode de pondération générique (gamme 0.15–0.40) |
| Concurrent change l'architecture cloud | Couvrir edge/mobile/cloud/hybride dans revendications systèmes |
| Concurrent utilise une autre blockchain | Revendiquer le résultat fonctionnel (immuabilité + horodatage certifié) |
| Concurrent intègre d'autres ERP | Revendiquer l'interface API générique, pas le connecteur spécifique |
